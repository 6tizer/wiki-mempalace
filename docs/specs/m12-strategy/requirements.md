# Requirements: M12 Strategy Suggestions

## Goal

- Produce explainable, read-only next-action suggestions from lint, gap, query,
  lifecycle, and metrics signals.
- Generate a durable suggestion report that can be read by agents and humans.

## Functional Requirements

- Add suggestion data model.
- Add at least 2 suggestion types:
  - `suggest.supersede_candidate`
  - `suggest.crystallize_candidate`
- Add optional `suggest.stale_review` if it can stay non-mutating.
- Add CLI command `wiki-cli suggest`.
- Output `code`, `severity`, `subject`, `reason`, `suggested_command`, and
  `execution_policy`.
- Support terminal text output.
- Support JSON output where JSON is the source of truth.
- Support Markdown output rendered from the same in-memory report for human
  reading.
- Support timestamped report files under `wiki/reports/suggestions/`.
- Default behavior must not mutate wiki state.

## Non-Goals

- No automatic supersede/crystallize execution.
- No LLM judge in first version.
- No internal operator/executor in first version.
- No deletion, discard, force promote, or cleanup action execution.
- No `--write-page` in first version.

## Inputs / Outputs

- Input: store, lint/gap findings, query/outbox history, metrics report.
- Output:
  - terminal text by default
  - optional JSON stdout via `--json`
  - optional timestamped JSON + Markdown report files via `--report-dir <PATH>`
- Default report directory: `wiki/reports/suggestions/`.
- Signal collection must use read-only paths. M12 must not call command paths
  that create audit/outbox events while collecting signals.
- `QueryServed` history must be scope-safe. The current event payload has
  `query_fingerprint` and `top_doc_ids` but no explicit scope; M12 must not emit
  raw query text or build crystallize commands from query history unless the
  event can be resolved as visible for the current `--viewer-scope`.

## Report Contract

- JSON is the canonical machine-readable source.
- Markdown is a human-readable view rendered from the same `StrategyReport`
  object.
- JSON and Markdown files must share one `report_id` and filename prefix.
- Markdown must identify its sibling JSON file as source of truth.
- Report filenames must preserve history, for example:
  - `2026-04-25T120000Z-m12-suggest.json`
  - `2026-04-25T120000Z-m12-suggest.md`
- Suggestions must include stable `suggestion_id` values within a report.
- `execution_policy` values:
  - `auto_safe`: action backed by current `FixActionType::Auto` behavior or an
    equivalent low-risk whitelist.
  - `agent_review`: non-deletion action that an operator agent may evaluate and
    execute later, including draft-style fixes and evidenced supersede or
    crystallize suggestions.
  - `human_required`: deletion, discard, force, disputed replacement, or
    insufficient-evidence action.

## Execution Boundary

- M12 only diagnoses and dispatches.
- M7 Fixer remains responsible for current low-risk auto fixes.
- M12 must treat current `FixActionType::Draft` and `FixActionType::Manual`
  outputs as non-auto suggestions.
- Future internal operator/executor may read the JSON report and produce a
  separate execution report.
- Execution reports are not generated by M12 first version.

## Acceptance Criteria

- At least 2 suggestion types tested.
- `--viewer-scope` enforced.
- Suggestions include reason and command.
- JSON and Markdown reports are generated from the same report data.
- Report files are timestamped and do not overwrite history.
- Default run does not write DB, outbox, or wiki projection.
- Query-history suggestions skip unresolved or hidden events rather than leaking
  raw query text.

## User / Agent Gates

- User approval needed: first suggestion rule set.
- Agent can automate: implementation/tests/docs/review.
