# Requirements: B5 Orphan Governance Follow-up

## Goal

- Upgrade B5 from a count-only report into a repeatable governance loop:
  audit real content, let LLM produce a plan, validate that plan, then let code
  apply only whitelisted actions.
- Fix the audit scope mistake from B5 v1: governance content is only
  `pages/` and `sources/`, not the whole vault root.

## Plain-Language Summary

- `vault-audit` is the measuring tool. It only measures the real shelves:
  `pages/` and `sources/`.
- `orphan-governance plan` is the foreman. It collects exact paths and asks the
  LLM for judgment only where rules are not enough.
- `orphan-governance apply` is the worker. It executes only known safe actions;
  the LLM never writes or deletes files directly.

## Functional Requirements

- `vault-audit`
  - Scan only `<vault>/pages` and `<vault>/sources` for content stats.
  - Do not classify `.wiki/`, `.obsidian/`, `_archive/`, `reports/`, root
    projections, or legacy root `concepts/` as content.
  - Write timestamped reports only:
    - `reports/vault-audit-<timestamp>.json`
    - `reports/vault-audit-<timestamp>.md`
  - Do not write `reports/vault-audit.json` or `reports/vault-audit.md`.
  - Include path-level lists for missing page `status`, missing source
    `compiled_to_wiki`, unsupported frontmatter, and orphan candidates.

- `orphan-governance plan`
  - Require a timestamped audit path. Reject undated `vault-audit.json`.
  - Gather source-to-summary title candidates from `sources/` and
    `pages/summary/`.
  - Add cleanup candidates only from the fixed whitelist below.
  - Use LLM through existing `--llm-config` / `complete_chat` infrastructure.
  - LLM output must be JSON and schema-validated before any report is written.
  - LLM may classify, explain, and recommend whitelisted actions only.
  - LLM must not invent paths, counts, files, or commands.
  - Write:
    - `reports/orphan-governance-plan-<timestamp>.json`
    - `reports/orphan-governance-plan-<timestamp>.md`
  - Markdown is Chinese, generated by LLM from the validated plan JSON.

- `orphan-governance apply`
  - Input is a governance plan JSON.
  - Default is dry-run.
  - `--apply` is required for mutation.
  - Allowed mutations:
    - insert missing `status` in files under `pages/`
    - insert missing `compiled_to_wiki` in files under `sources/`
    - remove fixed cleanup whitelist files/directories
  - Disallowed mutations:
    - edit body text
    - edit paths not present in the validated plan
    - delete non-whitelist files
    - run `batch-ingest`
    - write DB/outbox/palace

## Cleanup Whitelist

May be removed by apply:

- `.DS_Store`
- `concepts/7433d289.md`
- `concepts/`
- `_archive/legacy-root/AGENTS md 5da673ca2377484498ec12f5679bfbf3.md`
- `_archive/legacy-root/concepts/04ff4434.md`
- `_archive/legacy-root/concepts/`
- `_archive/legacy-root/analyses/`
- `.wiki/orphan-audit-report.md`
- `reports/vault-audit.json`
- `reports/vault-audit.md`

Must not be removed by this module:

- `_archive/legacy-root/README.md`
- `.wiki/wiki.db`
- `.wiki/palace.db`
- `.obsidian/`
- `pages/`
- `sources/`

## Root Concepts Bug

- Investigate why root `concepts/` reappeared after legacy-root archive.
- Fix any repository code path that can write root `concepts/`.
- Add regression coverage proving projection/sync paths do not write root
  `concepts/`.

## Acceptance Criteria

- `vault-audit` against a vault with `_archive`, `.wiki`, `reports`, and root
  `concepts` does not count those directories as content.
- `vault-audit` writes only timestamped audit files.
- `orphan-governance plan` rejects `vault-audit.json`.
- LLM plan JSON is validated and cannot reference paths outside audit evidence
  or cleanup whitelist.
- LLM-generated Markdown is Chinese and derived from validated plan JSON.
- `orphan-governance apply` dry-run does not mutate files.
- `orphan-governance apply --apply` mutates only whitelisted frontmatter fields
  and cleanup paths.
- No command in this module runs `batch-ingest`.
